Optimize Community Schema query latency

Composite persistent index on (partition_id, level) with storedValues: occurrence in graphrag_importer. Decision: ship solution B (= A2). Validated on ArangoDB 3.12.4.

Semi-live +1.34×–1.50× Press → or Space

Problem

Hot AQL path in the retriever

Filters by partition + level, returns three small fields — but Community docs are fat.

  • Before: importer only created partition_id_index
  • Gap: no level in key; no covering occurrence
  • Ownership: latency in retriever; indexes must be created by importer
// Community Schema query
FOR c IN @@communities_collection
  FILTER c.partition_id IN @partition_ids
  FILTER c.level <= @level
  RETURN {
    _key: c._key,
    level: c.level,
    occurrence: c.occurrence
  }

Proposed solution

Composite + storedValues

db._collection("…_Communities").ensureIndex({
  type: "persistent",
  fields: ["partition_id", "level"],
  storedValues: ["occurrence"],
  name: "partition_id_level_index",
  inBackground: true
});
  • fields order matches IN then <=
  • storedValues enables covering-style reads
  • inBackground avoids blocking writes during build
  • Replace lone Communities partition_id_index
